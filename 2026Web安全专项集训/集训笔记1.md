### 1.1 Web安全概括

#### 1.1.1 什么是Web安全

**Web 安全**是保护网络、服务器、用户、设备和 IT 系统免受基于互联网的网络攻击的学科。这其中包括恶意软件、网络钓鱼等攻击媒介，以及其他可能导致数据泄露和数据盗窃的攻击。Web 安全通常需要多层技术和防御措施，其中可能包括 Web 应用程序防火墙、入侵防御系统 (IPS)、URL 过滤和防病毒 / 反恶意软件解决方案。Web 安全可能包括 Web 应用程序和 API 安全，以及旨在保护云环境的云安全解决方案。

#### 1.1.2 Web应用和Web安全

Web 应用程序的兴起为人们提供了便捷的互联网服务，也带来了新的安全挑战。随着 Web 技术的不断演进，Web 安全经历了从 Web1.0 到 Web3.0 的演变过程，每个发展阶段都面临着独特的安全威胁和挑战。
##### Web1.0 时代（1990-2004）
Web1.0 时代是互联网的初始阶段，此阶段主要是**静态网页**的时代，网站内容以文本、图像和少量的多媒体内容为主。Web 应用程序的交互性和动态性相对较低，主要是展示性的信息发布。由于互联网的规模相对较小，Web 安全问题主要集中在服务端，例如对服务器进行入侵、拒绝服务攻击等。
##### Web2.0 时代（2004-2020）
Web2.0 时代标志着互联网的新阶段，网站的**交互式和动态化特征**显著增强，用户可以参与内容创作和分享。Web 2.0 时代涌现了大量社交媒体、博客、维基百科等应用，用户生成内容（User-Generated Content, UGC）成为主流。在 Web 2.0 时代，Web 应用程序的攻击面变得更加多样化和复杂化，新的安全威胁应运而生，如 XSS、CSRF 和 SQL 注入等。这些漏洞主要通过用户输入攻击 Web 应用程序，而不仅是直接侵害服务端。
##### Web3.0 时代（2020 - 至今）
目前，我们正逐渐步入 Web3.0 时代，Web3.0 作为未来互联网的发展方向，仍处于探索和发展阶段。Web3.0 被认为是**智能化和语义化**的互联网时代，将涉及人工智能、大数据、物联网等新技术的应用。此阶段的主要安全威胁有 API 安全、云安全、物联网安全等。
随着 Web3.0 的发展，Web 应用程序的安全挑战可能会更加复杂，涉及更多的智能化攻击和数据隐私问题。与 Web2.0 相比，Web3.0 可能会引入更多的安全性和隐私性保护措施，例如，引入区块链技术以确保数据的不可篡改性和用户身份的安全性。


#### 1.1.3 Web安全面临的挑战

Web 安全面临的主要威胁可归纳为以下几大类，其中多项已被列入 OWASP 十大安全风险，代表了当前最危险的网络威胁与漏洞：
1. 恶意软件（Malware）网络犯罪分子通过在用户终端或服务器上部署恶意程序，实施窃取凭据、发起拒绝服务攻击、资金转移等恶意行为。其中，勒索软件（Ransomware）危害尤为严重，通过加密受害者文件并索要赎金来换取解密密钥，造成巨大经济损失。
2. 钓鱼攻击（Phishing）攻击者利用伪造网站、电子邮件或其他欺骗性手段，诱导受害者主动泄露敏感信息（如登录凭证、支付卡信息等），属于社会工程学攻击的典型形式。
3. 网页 / 浏览器劫持（Web/Browser Hijacking）攻击者在用户不知情的情况下篡改浏览器设置，将其重定向至恶意网站，或植入键盘记录器、信息窃取程序等恶意组件，实现隐蔽的数据窃取。
4. 高级持续性威胁（APT）此类攻击具有高度隐蔽性和持续性，攻击者长期潜伏于企业 IT 环境中，通过持续监控、数据收集和操作观察，逐步渗透并破坏企业的安全体系与业务运营。
5. 注入攻击（Injection Attacks）攻击者利用 Web 应用输入验证机制的缺陷，通过表单、URL 参数等输入字段注入恶意代码（如 SQL 注入、命令注入等），从而篡改应用程序逻辑、获取未授权的系统访问权限或操纵数据库内容。
6. 拒绝服务攻击（DoS/DDoS）
通过向目标 IT 资源（如服务器、网络设备）发送海量恶意或虚假请求，耗尽其计算资源或网络带宽，导致服务瘫痪或性能严重下降，使合法用户无法正常访问服务。
7. 跨站脚本攻击（XSS）
攻击者利用 Web 应用对用户输入验证不足的漏洞，将恶意脚本注入网页并反射至其他用户浏览器中执行。XSS 攻击可用于窃取会话 Cookie、劫持用户会话、实施钓鱼攻击或以用户身份执行未授权操作。
8. 凭据泄露与账户接管（Credential Theft & Account Takeover）
攻击者利用窃取、泄露或暴力破解获得的登录凭据，非法访问用户账户，进而实施资金盗取、数据窃取、账户劫持或横向渗透企业 IT 环境等恶意行为。

##### 未来趋势
Web 安全是一个不断演变的领域，其发展过程伴随着层出不穷的新威胁与新挑战。面对攻击者不断升级的攻击手段，安全人员需要采取更具创新性的防御策略以应对。以下是 Web 安全的一些未来发展趋势。
🔒 云安全
在数字化转型加速的今天，各类组织如企业、政府机构等将大量数据和应用部署至云平台，云安全变得尤为重要。使用专为云环境设计的安全工具和策略，例如通过云访问安全代理（CASB）、数据加密、强身份验证和端点保护等方法，可以有效保护云环境中的 Web 应用程序不受威胁。
🔒 API 安全
随着 API（应用程序编程接口）在现代架构中扮演着越来越核心的角色，API 安全保护措施亟待加强。API 带来便捷的同时，其安全风险也不容小觑，尤其是身份认证和授权方面。各类组织在设计 API 时，应确保遵循安全领域的最佳实践，如使用 OAuth 等标准化认证授权机制，以及部署 API 网关进行流量管理与监控。此外，各类组织需定期执行 API 安全测试，确保 API 能够抵御恶意攻击。
🔒 供应链攻击
软件供应链环节的漏洞可导致严重的安全后果。各类组织必须对第三方库和组件的安全性进行审核，防范潜在的供应链攻击。供应链安全的关键在于实施全面的安全审核流程和策略，以确保引入的代码经过严格的安全校验，并持续监测第三方组件以及时发现和响应安全事件。
🔒 AI 安全
AI（人工智能）的引入为网络安全领域带来革命性进步，但未经加固的 AI 系统也可能受到攻击，例如，对抗性攻击可能通过输入精心设计的数据迷惑 AI 模型。因此，各类组织需要采取措施增强 AI 系统的安全性，通过数据清洗、模型硬化和持续的安全测试，以提高 AI 对抗这些复杂攻击手段的能力。
🔒 零信任安全架构
零信任安全模型基于 “永不信任，总是验证” 的原则，强调对无论来自内部还是外部的所有请求，都进行全面且持续的身份验证、授权和加密。该架构通过将访问权限限制在最小必要范围内，并对所有设备和用户进行持续的安全检查，从而有效降低数据泄露、恶意内部行为及其他网络攻击的风险。

### 1.2 Web应用架构

#### 1.2.1 什么是Web应用架构

Web 应用架构是指构建 Web 应用程序时所采用的整体设计结构和组件组织方式。它定义了不同组件之间如何交互，数据如何流动，以及系统如何响应用户请求。
##### Web 应用的基础组成
Web 应用的基本组成
🖥️ 前端 (Frontend)用户直接接触的部分，包括页面布局、交互效果、用户体验等。通常使用 HTML、CSS、JavaScript 等技术。
⚙️ 后端 (Backend)处理业务逻辑、数据处理、用户认证等核心功能。常用语言包括 Python、Java、Node.js 等。
🗄️ 数据库 (Database)存储和管理应用程序的数据，如用户信息、业务数据等。包括关系型和非关系型数据库。


#### 1.2.2 前后端分工详解

在 Web 应用开发体系中，前后端分工遵循 "关注点分离" 原则：前端（Frontend）聚焦于**客户端交互层**，负责页面结构构建（HTML）、样式渲染（CSS）、用户行为响应及数据可视化，并通过 Ajax/Fetch 等技术调用后端接口实现动态数据展示；后端（Backend）则专注于**服务端业务逻辑层**，承担请求处理、权限校验、核心算法执行、数据库交互及 API 接口封装等任务，确保数据处理的准确性与系统安全性。两者通过标准化的 HTTP 协议与数据格式（如 JSON）进行解耦协作，前端保障用户体验与交互流畅性，后端保障业务规则与数据可靠性，而在渗透测试视角下，这种分工边界也意味着需分别关注客户端代码审计（如 XSS、逻辑绕过）与服务端漏洞挖掘（如注入、越权），从而构建纵深防御的安全评估体系。

前端职责 (Frontend Responsibilities)
- 用户界面设计
负责页面布局、样式设计、交互效果，确保良好的用户体验。使用 HTML、CSS、JavaScript 等技术。
- 响应式设计
确保应用在不同设备和屏幕尺寸上都能正常显示和使用，提供一致的用户体验。
- API 集成
调用后端提供的 API 接口，获取数据并在前端展示，处理用户的交互操作。

后端职责 (Backend Responsibilities)
- 业务逻辑处理
处理核心业务规则、数据验证、权限控制等，确保数据的正确性和安全性。
- 数据管理
管理数据库连接、执行 CRUD 操作、数据备份恢复、性能优化等。
- 安全控制
用户认证、权限验证、数据加密、防止 SQL 注入等安全措施的实现。

#### 1.2.3 前后端通信方式

Web 应用的前后端通信就像 “前台与后台” 的协作：前端负责界面展示和接收用户操作，通过 HTTP 协议向后端发送请求；后端负责处理业务逻辑和查询数据库，再将结果返回给前端更新页面。双方使用 JSON 等通用格式交换数据，并通过 Cookie 或 Token 确认用户身份，确保传输安全，从而实现客户端与服务端的高效配合。

### 1.3 web协议解析

#### 1.3.1 URL
URL（Uniform Resource Locator，统一资源定位符）是一种用于定位互联网资源的地址，可以指向网页、图像、视频等各种类型的资源。作为互联网信息检索的关键组成部分，URL 为访问和获取网络资源提供了标准化的方法。

URL scheme://[username:password@]hostname[:port]/path[?query][#fragment]

- scheme：协议，如 http、https、ftp、file 等。
- username/password：密码。*身份认证信息，通常省略*
- hostname：域名或 IP 地址。*会被DNS解析成网络层的IP地址*
- port：端口号。*用于传输层的具体应用接口的确定*
- path：网页的路径。*定位到服务器上的资源路径*
- query：查询字符串，用于指定网页的请求参数。*用于传递请求数据*
- fragment：片段标识符，用于指定网页中的某个位置。*此标签不会被用于构造http请求，而保留在浏览器中，可能用于页面js的某些操作(可能产生XSS DOM型漏洞)*

#### 1.3.3 DNS 解析

我们访问某个网站，并不会直接使用它的 IP 地址，而是使用域名。例如：www.baidu.com，在网络中进行通信使用的是 IP 地址，因此需要把这个网址转换成对应的 IP 地址，这个过程就是 DNS 解析。DNS 解析的过程就是寻找哪台机器上有你需要资源的过程。DNS 存在着**多级缓存**，从离浏览器的距离排序的话，有以下几种：浏览器缓存，系统缓存，路由器缓存，IPS 服务器缓存，根域名服务器缓存，顶级域名服务器缓存，主域名服务器缓存。查询的优先级依次降低，也就是说刚才访问过的 URL 对应 IP 地址，被浏览器保存着，直接使用。如果找不到，就依照优先级由高到低依次查找。

- **作用**：将人类可读的域名（如 www.baidu.com）转换为机器可读的 IP 地址（如 192.0.2.1）。
- **过程**：
    1. 浏览器缓存：检查本地是否缓存过该域名的 IP。
    2. 系统缓存：查询操作系统（如 /etc/hosts 文件）。
    3. 路由器缓存：检查本地路由器或 ISP 的 DNS 缓存。
    4. 递归查询：若缓存未命中，向根域名服务器、顶级域服务器（如 .com）、权威域名服务器逐级查询。

#### 1.3.4 TCP 连接

一个典型的 URL 是以 http 或者 https 开头的，Web 应用程序是基于 HTTP 等协议实现通信的。HTTP 协议时使用 TCP 协议作为其传输层协议，因此会先发起 TCP 的 3 次握手，尝试和服务器 IP 及端口建立 TCP 连接。如果使用的是 HTTPS，在传输数据之前需要客户端与服务器进行一个握手 (TLS/SSL 握手)，在握手过程中将确立双方加密传输数据的密码信息。TLS/SSL 使用了非对称加密，对称加密以及 hash 等。相比于 HTTP，HTTPS 提供了安全保证，同时也会在时间上增加损耗。

![TCP 连接](./TCP三次握手流程.png)

三次握手：建立可靠的 TCP 连接
1. SYN：客户端发送同步序列号（SYN=1）到服务器。
2. SYN-ACK：服务器确认并回复自己的序列号（SYN=1, ACK=1）。
3. ACK：客户端确认服务器的序列号（ACK=1）

#### 1.3.5 发送 HTTP 请求

如果成功建立 TCP 连接，此时浏览器将会发送 HTTP 请求。发送 HTTP 请求的过程就是构建 HTTP 请求报文并通过 TCP 协议中发送到服务器指定端口（HTTP 协议 80/8080，HTTPS 协议 443）。HTTP 请求报文是由三部分组成：请求行、请求报头和请求正文。

- **请求组成**：
    1. 请求行：方法（GET/POST）、路径（如 /index.html）、协议版本（HTTP/1.1 或 HTTP/2）。
    2. 请求头：附加信息（如 Host、User-Agent、Cookie）。
    3. 请求体：POST/PUT 请求时携带的数据（如 JSON 或表单数据）。

**HTTP请求方法**:
| 请求方法 | 描述 | 常用场景 |
| --- | --- | --- |
| GET | 用于从服务器获取数据，请求参数附加在URL中 | 获取资源的请求，如获取网页、图像等 |
| POST | 用于向服务器提交数据，请求参数包含在请求主体中 | 提交表单数据、创建新资源等操作 |
| PUT | 用于更新服务器中的资源，将请求数据存储在指定的URL。如果URL已存在，则更新资源；如果不存在，则创建新资源 | 更新资源的请求，如更新文本文件、图片等 |
| DELETE | 用于从服务器上删除指定的资源 | 删除资源的请求，如删除文件、记录等 |
| HEAD | 与GET类似，但服务器只返回响应的头部信息 | 检查资源是否存在，获取资源的元信息等 |
| OPTIONS | 用于请求关于目标资源的通信选项。服务器告知客户端允许的请求方法、头部字段等 | 获取目标资源的通信选项，如跨域资源共享等 |
| CONNECT | 用于建立网络连接，通常用于代理服务器，指示服务器与客户端建立直接连接以进行隧道通信 | 建立网络连接，通常用于代理服务器 |

#### 1.3.6 服务器处理请求并返回

Web 服务器接收到 TCP 报文后，会对 TCP 连接进行处理，对 HTTP 协议进行解析，读取 HTTP 请求报文信息，根据协议规范进行报文解析，然后向浏览器返回响应报文。HTTP 请求的如果是静态网页，直接返回网页信息，如果是动态页面则还需要对数据库进行查询，才能得到完整的页面信息。HTTP 响应报文也是由三部分组成：状态码、响应报头和响应报文。响应报文中是服务器返回给浏览器的文本信息，包括：HTML、CSS、JS、图片等。

- **后端流程**：
  1.  Web 服务器（如 Nginx/Apache）：接收请求，转发到应用服务器（如 Node.js、Tomcat）。
  2.  应用逻辑：执行业务代码（如数据库查询、API 调用）。
  3.  响应生成：返回 HTML、JSON 或其他格式数据。

- **响应报文**：
  1.  状态行：状态码（如 200 OK、404 Not Found）。
  2.  响应头：Content-Type、Cache-Control 等元数据。
  3.  响应体：实际内容（如 HTML 或图片数据）。

![HTTP响应报文](./HTTP响应报文.png)

 HTTP 状态码分类及说明
 ✅ 2xx 成功
- **200 OK**：请求成功
- **201 Created**：资源创建成功
- **204 No Content**：请求成功但无内容
✅ 3xx 重定向
- **301 Moved Permanently**：永久重定向

#### 1.3.7 浏览器解析渲染页面

浏览器重现页面效果是一个**边解析边渲染**的过程，首先解析 HTML 文件构建 DOM 树，然后解析 CSS 文件构建渲染树，并在屏幕上显示页面效果。JS 的解析是由浏览器中的 JS 解析引擎完成的。浏览器在解析过程中，如遇到请求外部的图像、iconfont、JS 等资源时，浏览器将重复以上过程下载该资源。

HTTP 协议运行的过程就是：客户端发起请求和服务器请响应，服务器响应后，服务器并不会记录客户端的信息，每次请求都是独立的，也就是说 HTTP 是无状态的。服务器被动接收请求，对客户端情况一无所知，因此从安全角度来看客户端的所有行为都不可信。在静态网页的时代，HTTP 协议简单、快速、灵活性强，能够高效完成网页访问。但是在动态网页时代，页面和服务器的交互增加，如果每次都要重传所有信息，效率很低。因此引入了 cookie 和 session 等机制实现有状态的 Web。需要注意的是不仅仅是浏览器可以发起，也可以在各种应用程序中发起。例如使用 curl 命令行工具在 kali 系统命令行中执行效果如图

#### 1.3.8 会话管理

由于 HTTP 是无状态的，即每次请求和响应相互独立，服务器无法识别不同的客户端，也无法保存客户端的状态信息。为解决 Web 应用程序中客户端与服务器之间通信的连续性问题，会话技术被广泛采用。
会话是指在用户与网站或用户与 Web 应用程序之间建立的持久性连接，用于跟踪用户的状态和身份，每个会话都有一个唯一的标识符，通常称为会话 ID。会话 ID 一般存储在用户的浏览器中，通常以 Cookie 或 URL 参数的形式出现，用于标识特定用户的会话。会话在许多 Web 应用程序中用于维护用户的登录状态、存储临时数据以及确保用户与 Web 应用程序之间交互的连续性。简单来说，会话可以理解为用户打开一个浏览器，访问网站进行一系列交互操作（如点击多个链接、提交表单等），并且在这些操作过程中，服务器能够通过会话 ID 识别并保持用户的状态。
会话技术主要分为两大类：客户端会话技术（如 Cookie 机制）和服务端会话技术（如 Session 机制）。
##### Cookie机制

Cookie（也称作Web Cookie或浏览器Cookie）是由服务器生成并发送到用户浏览器的小型信息文件，由用户的计算机进行保存。客户端浏览器通常会设定Cookie的最长保存时间，一旦超过这个时间，系统会自动将其清除。Cookie主要用于跟踪用户的会话状态、存储用户的个人偏好设置以及执行其他与用户相关的功能。

- **Cookie的基本格式如下**：

```
<name>=<value> [;Expires=<date>] [;Domain=<domain_name>]
[;Path=<path>] [;Secure] [;HttpOnly]
```
其中，<name>表示Cookie名称，由字母、数字、下划线和减号组成，不能包含空格或特殊字符。<value>表示Cookie值，可以是任何字符串。属性是可选的，包括以下部分。
(1) Expires: 指定Cookie的过期时间，以GMT（格林尼治标准时间）或UTC（协调世界时）时间格式表示。
(2) Domain: 指定Cookie的有效域名。
(3) Path: 指定Cookie的有效路径。
(4) Secure: 指定Cookie仅在HTTPS连接上发送。
(5) HttpOnly: 指定Cookie仅在HTTP请求中发送，不允许通过JavaScript访问。
![Cookie示例](./Cookie举例.png)

- **Cookie的工作流程**：
1. 当浏览器首次请求目标服务器时，不会携带任何Cookie，服务器收到请求后，通过响应报文中的Set-Cookie字段设置一个新的Cookie。
2. 浏览器解析响应中的Set-Cookie字段，提取响应中的Cookie并对其进行保存。
3. 接着，浏览器在向服务器发送的每个请求中都会携带之前存储的Cookie，服务器收到请求后会解析请求中的Cookie字段，提取并处理其中的Cookie。
4. 如果需要生成新的Cookie，服务器会在响应中设置Set-Cookie字段，其中包含更新后的Cookie。
5. 然后，服务器将响应发送回浏览器，浏览器解析响应中的Set-Cookie字段，并存储新的Cookie，以便在将来的请求中使用。工作流程如图所示。

- **Cookie的销毁**：
关于Cookie的销毁：如果Cookie设置了Expires或Max-Age属性，当到达设定的过期时间后，浏览器会自动删除该Cookie；若Cookie没有设置过期时间，它将在浏览器关闭时被销毁。用户也可以通过浏览器手动清理Cookie，从而立即删除相关信息。

- **Cookie的局限性**：

（1）大小和数量限制：Cookie的大小和数量都存在限制。每个域名下存储的Cookie数量通常被限制为50个左右，每个Cookie的大小限制为4KB左右，这在一定程度上限制了Cookie存储大量数据的能力。

（2）隐私问题：Cookie能够用于追踪用户的浏览活动，这可能会引发用户对隐私的担忧，特别是第三方Cookie在网络追踪和个人隐私方面引发了广泛争议。因此，许多浏览器提供了隐私模式，以限制Cookie的存储和使用。

（3）安全问题：由于Cookie可能存储敏感信息（如会话ID），如果不采取适当的保护措施，攻击者可能会利用这些信息进行攻击，如跨站脚本（XSS）攻击或跨站请求伪造（CSRF）攻击。
- **两种Cookie**:
| 类型     | Session Cookie                                                                 | Persistent Cookie                                                                 |
|----------|--------------------------------------------------------------------------------|----------------------------------------------------------------------------------|
| 生命周期 | 当用户关闭浏览器时会被删除                                                       | 在用户关闭浏览器后仍然存在，直到达到指定的过期日期                                 |
| 存储位置 | 存储在内存中，不会写入磁盘                                                       | 写入磁盘，占用用户设备中的空间                                                     |
| 安全性   | 通常较安全，因为只在单个会话期间存在，不易被持久保存的攻击威胁利用                 | 需要更多的安全措施，因其在浏览器关闭后仍然存在，容易受到更长时间的攻击威胁           |
| 用途     | 在单个浏览会话期间维持用户在网站中的活动信息                                     | 存储需要在多个浏览会话中访问的信息                                                 |
##### Session机制
Session是一种服务端会话管理机制，用于跟踪用户状态和存储用户数据。通过在服务端保存数据，Session可以保持用户的登录状态、存储购物车内容和用户行为数据等，以提供个性化的用户体验。Session机制和Cookie机制具有相似的用途，根据数据存放位置进行区分：Session机制是服务端会话技术，数据保存在服务端；Cookie机制是客户端会话技术，数据保存在客户端。

Session的具体格式和存储方式取决于后端编程语言和框架的实现，一般以键值对的形式进行存储。以PHP中的Session文件为例，文件名格式一般为“sess_<sessionid>”，例如“sess_eaccd21635f8f2e19632f02300d323d0”，并对其反序列化后，如图所示。
![Session机制](./Session例子.png)

- **Session的工作流程**：
1. 当用户首次在浏览器中访问一个网站时，浏览器会向该网站的服务器发送一个HTTP请求。
2. 服务器如果检测到用户没有现有的Session记录，将会创建一个新的Session并存储在服务器，由Session ID唯一标识该用户的Session。
3. 服务器将生成的Session ID通过HTTP响应报文中的Set-Cookie字段发送给用户的浏览器，浏览器将此Session ID存储为一个Cookie，在后续访问网站的请求中，浏览器会自动携带该Cookie发送给服务器。
4. 服务器接收到请求后，提取出Session ID并查找对应的Session记录。如果该Session经验证是有效的，服务器便能识别用户并恢复之前的状态。
- **Session的局限性**：

（1）生命周期：Session通常设有一个过期时间，即用户在一段时间内无活动后，会话将被视为过期并被清除。过期时间的设定需要在用户体验和安全性之间进行权衡，过短可能导致用户频繁重新登录，过长则可能增加安全风险。

（2）扩展性问题：当应用需要在多个服务器之间共享Session时，实现起来较为复杂，因为Session默认仅存储在单个服务器中，当用户的请求被路由到不同的服务器上时，需要额外的机制共享这些Session数据，如通过数据库或分布式缓存等方式，这增加了系统的复杂度和开销。

（3）资源消耗：Session信息存储在服务器中，随着在线用户数量的增加，所占用的服务器存储空间也会相应增长，这对服务器资源是一种挑战，尤其在资源有限的环境中。

##### Cookie和Session的对比

| 类型     | Cookie                                                                 | Session                                                                 |
|----------|------------------------------------------------------------------------|-------------------------------------------------------------------------|
| 存储位置 | 存储在客户端的浏览器中                                                   | 存储在服务端的内存或持久化介质中                                           |
| 数据类型 | 通常只能存储字符串类型数据                                               | 可以存储复杂的数据结构，如对象、数组等                                     |
| 安全性   | 相对较低，容易被篡改                                                     | 相对较高，不易被窃取或篡改                                                 |
| 大小限制 | 受浏览器限制，通常在4KB左右                                              | 理论上无大小限制，但受服务器配置和性能限制                                 |
| 生命周期 | 指定明确的过期时间，无论用户在该时间内是否有进一步的操作，最终都会被销毁     | 基于用户活动间隔计时，如果用户在设定时间内无操作则自动销毁；用户的每次访问都会重置计时 |
| 跨域发送 | 受同源策略限制，只能在同一域下使用                                       | 不受同源策略限制，通过Session ID在不同域间传递                             |
| 传递机制 | 通过HTTP头部传递，存储在客户端                                           | 通过Session ID在客户端和服务器之间传递                                     |
| 用途     | 跟踪用户状态，存储少量信息，如用户偏好设置                               | 跟踪用户会话状态，存储更丰富的用户相关信息                                 |

#### 1.6.5 BurpSuite 常用模块介绍

1.  Proxy（代理）
    - 核心功能：拦截浏览器与服务器之间的 HTTP/HTTPS 请求和响应。
    - 用途：修改请求参数（如 Cookie、表单数据）、重放请求、测试漏洞（如 SQL 注入、XSS）。
    - 特点：支持 HTTPS 解密、自动修改请求头（如 User-Agent）。

2.  Repeater（重放器）
    - 功能：手动修改并重新发送请求，观察服务器响应。
    - 用途：测试输入验证、逻辑漏洞（如越权访问）、API 安全。

3.  Intruder（入侵者）
    - 功能：自动化暴力破解、模糊测试（Fuzzing）、参数枚举。
    - 用途：爆破登录表单、测试 IDOR（不安全的直接对象引用）、检测 SQL 注入。

4.  Scanner（扫描器，仅限 Pro 版）
    - 功能：自动化扫描 Web 应用漏洞，如 SQL 注入、XSS、CSRF、SSRF、文件包含。
    - 特点：支持主动扫描（发送攻击载荷）和被动扫描（仅分析流量）。

5.  Decoder（解码器）
    - 功能：对 Base64、URL、Hex、HTML、JSON 等编码数据进行解码/编码。
    - 用途：分析加密数据、绕过 WAF（Web 应用防火墙）。

6.  Comparer（比较器）
    - 功能：对比两个 HTTP 响应的差异（如不同权限用户的响应）。
    - 用途：检测信息泄露、越权漏洞。

7.  Extender（扩展）
    - 功能：支持 Python、Java 插件扩展，如 SQLMap 集成、自定义漏洞检测。可暂停请求/响应，修改内容（如参数、头信息）后再放行。将拦截的请求发送到Repeater模块，手动修改参数（如SQL注入、XSS测试）并重复发送，观察服务器响应。
